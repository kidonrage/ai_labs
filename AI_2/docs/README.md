# Документация проекта

## Что это

Это учебный проект локального чат-интерфейса для работы с LLM без сборщика и без тяжелой инфраструктуры. Основной контур написан на `HTML + CSS + JS modules`, хранит состояние в `localStorage`, умеет работать с несколькими чатами, профилями пользователя, инвариантами поведения агента и режимом `RAG`.

Проект собран по принципу "минимум магии":

- фронтенд запускается как обычная статическая страница;
- бизнес-логика лежит в `src/`;
- состояние живет в браузере;
- RAG использует готовые JSON-индексы из `static/`;
- рядом лежат отдельные учебные Python/MCP-сервисы, которые не обязательны для базового запуска UI.

## Основные возможности

- несколько независимых чатов с переключением в боковой панели;
- создание нового чата из текущего состояния;
- профили пользователя с собственным стилем ответа;
- инварианты, которые ограничивают поведение агента;
- пошаговое управление задачей через task workflow;
- поддержка нескольких API-режимов:
  - `ProxyAPI Responses`;
  - `Ollama Chat`;
  - `Remote Ollama`;
  - `Ollama + MCP`;
- режим `RAG` с локальным индексом и эмбеддингами через Ollama;
- batch/test-инструменты для проверки конфигураций LLM и режимов RAG;
- восстановление состояния после перезагрузки страницы.

## Архитектура

### 1. UI-слой

- Точка входа: `index.html`
- Bootstrap приложения: `src/app.js`
- Контроллер верхнего уровня: `src/app/app-controller.js`

UI не требует сборщика. `index.html` подключает стили и ES-модули напрямую. Вся инициализация идет через `AppController.bootstrap()`.

### 2. Слой приложения

Каталог `src/app/` отвечает за прикладную координацию:

- `app-controller.js` связывает UI, репозиторий, сессию чата и настройки;
- `chat-session-controller.js` управляет активным агентом и жизненным циклом чата;
- `composer-controller.js` обрабатывает отправку сообщений и служебные команды;
- `chat-repository.js` нормализует и сохраняет рабочее пространство;
- `workspace-store.js` управляет чатами, профилями и активным состоянием;
- `llm-config-test-controller.js` запускает тест конфигураций моделей;
- `rag-config.js` синхронизирует RAG-настройки между UI и агентом.

### 3. Агент

Основная модель поведения лежит в `src/agent.js` и подпапке `src/agent/`.

Ключевые части:

- `model-gateway.js` делает HTTP-вызовы к LLM API;
- `memory-update-service.js` обновляет память агента;
- `context-builders.js` собирает контекст для модели;
- `invariant-guard.js` проверяет конфликт запроса с инвариантами;
- `state-serializer.js` экспортирует и импортирует состояние агента;
- `rag-answer-service.js` отвечает за режим ответа с RAG.

Агент поддерживает:

- историю сообщений;
- краткосрочную и долгосрочную память;
- task state;
- пользовательский профиль;
- RAG-конфиг;
- проверку инвариантов до ответа модели.

### 4. RAG

RAG-контур лежит в `src/rag/` и файлах `src/rag.js`, `src/rag-batch.js`.

Что используется по умолчанию:

- индекс: `./static/index_structured.json`;
- endpoint эмбеддингов: `http://localhost:11434/api/embed`;
- embedding model: `embeddinggemma`.

Поддерживаются режимы:

- `baseline`;
- `rewrite_only`;
- `filter_only`;
- `rewrite_and_filter`.

Если индекс недоступен по HTTP, приложение дает диагностическое сообщение, а не молча ломается.

### 5. Хранение состояния

Состояние хранится в браузере через `localStorage`.

- модуль хранения: `src/storage.js`;
- ключ хранения: `simple_agent_chat_v2`.

Сохраняются:

- список чатов;
- активный чат;
- ветки состояний;
- профили;
- сериализованное состояние агента.

Это удобно для локальной разработки, но не является серверным хранилищем и не защищает данные от очистки браузера.

## Что является основным контуром

Для повседневной работы важны:

- `index.html`;
- `src/`;
- `static/`;
- `src/config/private.config.example.js`.

Остальное вокруг них является вспомогательным или учебным контуром.

## Что является вспомогательным

### `project/`

Отдельный учебный Python-скрипт, который строит RAG-индексы по локальным документам и записывает `index_fixed.json` и `index_structured.json`.

### `search_server.py`, `summarize_server.py`, `save_server.py`, `git_branch_server.py`, `pipeline_server.py`

Простые демонстрационные MCP/FastMCP-сервисы:

- поиск по локальному словарю;
- примитивная суммаризация;
- сохранение результата в файл;
- получение имени текущей git-ветки проекта;
- оркестрация этих шагов в пайплайн.

### `simple_pipeline_mcp/`

Самодостаточный вариант простого MCP-пайплайна в одном сервере.

### `periodic_summary_mcp/`

Учебный MCP-сервис с фоновым воркером и периодическим обновлением summary.

### `weather_mcp/`

Тестовый mock weather MCP server.

Эти каталоги полезны для экспериментов, но не требуются для запуска основного чата.

## Требования

Минимум:

- современный браузер с поддержкой ES modules;
- `python3` для локального статического сервера;
- `node` для запуска тестов.

Опционально:

- `Ollama`, если нужен локальный LLM или RAG;
- доступ к ProxyAPI, если нужен внешний провайдер;
- `git`, если вы хотите запускать MCP tool `get_git_branch`;
- Python-библиотека `mcp`, если вы хотите запускать MCP-демо-сервисы.

## Быстрый старт

### 1. Запустить статический сервер

Из корня проекта:

```bash
python3 -m http.server 8080
```

Потом открыть в браузере:

```text
http://localhost:8080
```

Важно: открывать `index.html` напрямую как `file://` не стоит. RAG-индексы и модульный фронтенд надежнее работают через обычный HTTP.

### 2. Настроить приватный ключ при необходимости

Если используется внешний API, создайте файл:

```text
src/config/private.config.js
```

На основе шаблона:

```js
export const PRIVATE_APP_CONFIG = {
  apiKey: "sk-...",
  remoteOllama: {
    login: "your-login",
    password: "your-password",
  },
};
```

Файл `private.config.js` не должен попадать в публичный репозиторий.

### 3. Выбрать режим API в интерфейсе

В настройках чата доступны режимы:

- `ProxyAPI Responses` для внешнего OpenAI-совместимого endpoint;
- `Ollama Chat` для локального Ollama;
- `Remote Ollama` для удаленного Ollama;
- `Ollama + MCP` для сценариев с инструментами.

### Дополнительно: MCP server `get_git_branch`

Из корня проекта:

```bash
python3 ./git_branch_server.py
```

Сервер использует `stdio` transport и зарегистрирован в `mcp-config.json`.
Для ручной проверки через MCP-клиент вызовите tool `get_git_branch`.
При успехе сервер вернет структуру вида:

```json
{"branch": "feat/local_testing"}
```

## Запуск с Ollama

Если нужен локальный режим, поднимите Ollama и загрузите модели.

Пример:

```bash
ollama pull gemma3
ollama pull embeddinggemma
ollama serve
```

По умолчанию проект ожидает:

- чат API: `http://localhost:11434/api/chat`;
- embedding API: `http://localhost:11434/api/embed`.

## Как работает RAG

RAG в проекте включается для конкретного чата через настройки.

Базовый поток:

1. Пользователь включает `RAG`.
2. Фронтенд загружает индекс из `static/index_structured.json`.
3. Для вопроса строится embedding через Ollama endpoint.
4. Находятся релевантные чанки.
5. В зависимости от режима применяются rewrite/filter этапы.
6. Агент формирует ответ на базе найденного контекста.

Если релевантность слабая, проект старается вернуть безопасный ответ в духе "не знаю по имеющемуся контексту", а не выдумывать факты.

## Как пересобрать индексы RAG

Для этого используется каталог `project/`.

Пример запуска:

```bash
cd project
python3 main.py
```

Скрипт:

- читает документы из `project/docs/`;
- строит fixed и structured chunks;
- получает эмбеддинги через Ollama;
- сохраняет результат в `project/index_fixed.json` и `project/index_structured.json`;
- при наличии каталога `../static/` обновляет индексы и там.

## Тесты

Тесты лежат в `src/tests/` и запускаются напрямую через Node.

Прогон всех тестов:

```bash
find src/tests -name '*.test.mjs' -print0 | xargs -0 -n1 node
```

Отдельно есть структурная проверка, что `.js` и `.css` файлы в `src/` не превышают 300 строк. Это соответствует правилу проекта на ограничение размера файлов.

## На что обратить внимание

### Безопасность

- API-ключ не хранится в `localStorage`, он подгружается из `private.config.js`.
- Состояние чатов хранится локально в браузере и может содержать историю переписки.
- MCP-демо-сервисы предназначены для локального запуска и не выглядят как hardened production-контур.
- `save_server.py` пишет файл в текущую директорию, но принудительно режет путь до имени файла через `Path(filename).name`.

### Надежность

- приложение не зависит от сборщика и большого числа пакетов;
- при отсутствии сохраненного состояния поднимается дефолтный чат;
- нормализация состояния делает миграцию старых форматов менее болезненной;
- при ошибках загрузки RAG-индекса и при слабом контексте предусмотрены безопасные fallback-сценарии.

### Ограничения

- это локальный браузерный stateful UI, а не многопользовательская серверная система;
- история и профили завязаны на конкретный браузер;
- для полноценного RAG нужен локальный embedding endpoint;
- часть Python-каталогов в репозитории является учебной и не интегрирована в основной UI напрямую.

## Краткий вывод

Проект построен как локальная учебная лаборатория вокруг LLM-агента: без сборки, без тяжелого бэкенда, с упором на прозрачную архитектуру и воспроизводимость. Основной путь запуска простой: поднять статический сервер, при необходимости включить Ollama, открыть UI в браузере и работать с чатами, профилями, инвариантами и RAG.
